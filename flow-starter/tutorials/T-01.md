# Tutorial 01 — Why `for` / `while` / `if` Make Your Brain Work Harder

> **flow-arch / flow-starter**
> You finished tutorial-00 and understand pure functions.
> Now let's talk about the specific structures that fight against that thinking —
> and when they are still the right tool.

---

## The honest starting point

`for`, `while`, and `if` are not broken.
They are not mistakes.

They are the structures closest to how a CPU actually works:
conditions, jumps, loops.

But there is a specific problem with them — not with the syntax,
but with what they force your brain to do.

> **The real question is not "is `for` bad?"**
> **It is "what cognitive work does this code demand from the reader?"**

---

## 1. The cognitive burden of imperative control flow

When you read this:

```javascript
const result = [];

for (let i = 0; i < arr.length; i++) {
  if (arr[i] > 10) {
    result.push(arr[i] * 2);
  }
}
```

Your brain must simultaneously track:

| What you must remember              | Why it matters                              |
| ----------------------------------- | ------------------------------------------- |
| The loop counter `i`                | Is it starting at 0? Will it skip elements? |
| The exit condition `i < arr.length` | When does this stop?                        |
| The mutable `result` array          | What state is it in right now?              |
| The `push` operation                | Is this adding or replacing?                |
| The nesting level                   | Which `}` closes which block?               |

This is called **working memory load** — the number of things you must hold in your head at the same time to understand what the code does.

The code is not necessarily wrong. But to understand its _intent_, you have to mentally simulate its _execution_.

**That is the core problem.**

---

## 2. The declarative alternative — same result, less brain work

```javascript
const result = arr.filter((x) => x > 10).map((x) => x * 2);
```

What your brain needs to track now:

| What you read          | What it means                      |
| ---------------------- | ---------------------------------- |
| `.filter(x => x > 10)` | Keep only elements greater than 10 |
| `.map(x => x * 2)`     | Double each of them                |

That is it. Two thoughts. You read the _intent_ directly — no simulation required.

The data flows in one direction. Nothing is mutated. No index, no exit condition, no intermediate array to manage.

> **If code makes you simulate its execution to understand it, it is complex.**
> **If code lets you read its transformation directly, it is clear.**

---

## 3. The nesting problem — when `if` becomes a maze

`if` itself is not the enemy. The problem is **nesting + mutation + depth**.

### The maze version:

```javascript
if (user) {
  if (user.loggedIn) {
    if (user.role === "admin") {
      if (user.permissions.includes("write")) {
        doSomething();
      }
    }
  }
}
```

To understand this, you must:

- Read four conditions in sequence
- Track which branch you are in
- Remember what each level is checking
- Find where each `}` closes

This is the **pyramid of doom** — also called arrow code, because the indentation forms an arrow pointing right.

### The flow-arch alternative — guard clauses:

```javascript
if (!user?.loggedIn) return;
if (user.role !== "admin") return;
if (!user.permissions.includes("write")) return;

doSomething();
```

Now each line is a **single, independent decision**.
You read top to bottom. Each `return` eliminates a case immediately.
The happy path (`doSomething()`) is at the bottom, unindented, obvious.

**The logic is the same. The cognitive work is completely different.**

### The declarative version when conditions are data:

```javascript
const isAuthorized = (user) =>
  user?.loggedIn && user.role === "admin" && user.permissions.includes("write");

if (isAuthorized(user)) doSomething();
```

Now the authorization logic is a **pure function** — named, testable, reusable.
The `if` at the call site is a single condition, not a nested tree.

## 4. Why `while` is the hardest to read

`while` is a condition-driven loop. It keeps running until something changes.

```javascript
let i = 0;
while (i < items.length) {
  process(items[i]);
  i++;
}
```

To understand this loop, your brain must simulate **time**:

- What is `i` at the start?
- What happens to `i` inside?
- What is the exit condition?
- Could `i` be modified inside `process()`?
- Is there a risk of an infinite loop?

This is the **temporal burden** — you have to mentally run the loop forward to understand it.

### Why `while` is sometimes a legitimate tool

`while` is appropriate when:

```javascript
// Reading from a stream — you genuinely don't know when it ends
while ((line = readLine()) !== null) {
  process(line);
}

// Polling — waiting for a condition to become true
while (!server.isReady()) {
  wait(100);
}

// Recursive-like traversal — processing a linked list or tree
while (node !== null) {
  visit(node);
  node = node.next;
}
```

In these cases, `while` expresses "repeat until a condition changes" — which is exactly what the problem requires. The loop is not a style choice; it is the right model.

### The flow-arch position on `while`

In flow-arch, `while` lives at the **boundary** — in the infrastructure layer, not in the transformation layer.

```
Boundary (allowed):  while loop reading from a socket
                     polling for server readiness
                     animation frame loop

Transformation (not here):  .filter().map().reduce()
                             pure functions
                             no loops, no mutation
```

## 5. When `for` is still the right tool

This is the honest part. There are cases where a plain `for` loop is clearer than a chain of higher-order functions.

### Case 1 — Early exit from a search

```javascript
// Declarative — searches the entire array even after finding the answer
const found = items.map(transform).filter(isMatch)[0];

// Imperative — stops as soon as it finds the answer
let found;
for (const item of items) {
  if (isMatch(transform(item))) {
    found = item;
    break; // exit immediately
  }
}

// Better declarative — .find() has early exit built in
const found = items.find((item) => isMatch(transform(item)));
```

> **Rule:** Use `.find()` instead of `.filter()[0]`. It exits early.

### Case 2 — When the chain becomes unreadable

```javascript
// This is three layers of map + reduce — harder to read than a for loop
const result = items
  .map((x) => x.values)
  .reduce((a, b) => a.concat(b), [])
  .map((v) => v * factor)
  .reduce((sum, v) => sum + v, 0);

// A named for loop with clear intent is clearer here
let total = 0;
for (const item of items) {
  for (const value of item.values) {
    total += value * factor;
  }
}
```

The flow-arch principle is not "never use `for`". It is:

> **Use the structure that makes the intent most readable.**
> **For data transformation pipelines, declarative wins.**
> **For early exit, complex accumulation, or algorithmic logic — use the simplest structure.**

### Case 3 — Performance-critical inner loops

```javascript
// .map() creates a new array on every call — memory allocation
const result = hugeArray.map(double);

// A pre-allocated for loop avoids that allocation
const result = new Array(hugeArray.length);
for (let i = 0; i < hugeArray.length; i++) {
  result[i] = hugeArray[i] * 2;
}
```

In practice, this difference only matters at very large scales. For most application code, `.map()` is fast enough. But it is important to know the tradeoff exists.

## 6. The maturity model — a spectrum, not a binary

Most developers go through this progression:

```
Stage 1 — Beginner
  Write everything with for / if / while.
  It works. It's what you learned first.

Stage 2 — Discovery
  Discover map / filter / reduce.
  Feel like you've found something better.
  Start avoiding for / while entirely.
  Possibly over-engineer with chains.

Stage 3 — Dogma
  "for loops are bad"
  "OOP is wrong"
  "pure functions always"
  Strong opinions. Sometimes right. Often too absolute.

Stage 4 — Maturity
  Use the simplest structure for the problem.
  Sometimes that's .map().filter().
  Sometimes that's a for loop with a comment.
  The goal is readable intent, not stylistic purity.
```

flow-arch sits at **Stage 4** — with a clear preference for declarative style in transformation layers, and honest acknowledgment that imperative tools have their place at the boundary.

## 7. The flow-arch rule — where each structure belongs

```
┌─────────────────────────────────────────────────────┐
│  TRANSFORMATION LAYER  (pure, declarative)           │
│                                                     │
│  .filter()  .map()  .reduce()  .find()              │
│  pure functions, function composition               │
│  no mutation, no loops, no side effects             │
└─────────────────────────────────────────────────────┘
                         ↕
┌─────────────────────────────────────────────────────┐
│  BOUNDARY LAYER  (impure, but contained)             │
│                                                     │
│  for / while — infrastructure loops                 │
│  if — guard clauses at entry points                 │
│  DOM manipulation, fetch, logging                   │
│  Web Component connectedCallback                    │
└─────────────────────────────────────────────────────┘
```

**The transformation layer never has a `for` loop.**
**The boundary layer never has business logic.**

This separation is the entire point of flow-arch.

## 8. Practical examples in flow-arch style

### Bad — business logic inside a loop

```javascript
// ❌ Mixed concerns — transformation inside imperative loop
connectedCallback() {
  const result = [];
  for (let i = 0; i < this.state.items.length; i++) {
    if (this.state.items[i].active) {
      result.push(this.state.items[i].name.toUpperCase());
    }
  }
  this.shadowRoot.innerHTML = `<ul>${result.map(n => `<li>${n}</li>`).join('')}</ul>`;
}
```

### Good — transformation extracted as pure functions

```javascript
// ✅ Pure transformation functions
const isActive = (item) => item.active;
const toUpperName = (item) => item.name.toUpperCase();
const toListItem = (name) => `<li>${name}</li>`;

// ✅ Pure view function — declarative pipeline
const view = (state) => `
  <ul>
    ${state.items
      .filter(isActive)
      .map(toUpperName)
      .map(toListItem)
      .join('')}
  </ul>
`;

// ✅ Boundary — Web Component shell, side effects contained here
connectedCallback() {
  this.shadowRoot.innerHTML = view(this.state);
}
```

The `for` loop is gone — not because `for` is wrong, but because `.filter().map()` expresses the intent more clearly for this type of transformation.

## Summary

| Structure         | Problem                                                | When it's appropriate                                 |
| ----------------- | ------------------------------------------------------ | ----------------------------------------------------- |
| `for` (index)     | Tracks i, length, exit condition — high cognitive load | Performance-critical loops, pre-allocated arrays      |
| `for...of`        | Better than index, but still imperative                | Early exit with `break`, complex accumulation         |
| `while`           | Temporal burden — must simulate time                   | Stream reading, polling, tree traversal               |
| nested `if`       | Pyramid of doom — nesting obscures intent              | Never — refactor to guard clauses or pure predicates  |
| guard clause `if` | Single decision, clear intent                          | Entry point validation in boundary layer              |
| `.filter()`       | —                                                      | Keeping elements matching a condition                 |
| `.map()`          | —                                                      | Transforming every element                            |
| `.reduce()`       | Can become unreadable when chained                     | Accumulation when `.filter()` + `.map()` isn't enough |
| `.find()`         | —                                                      | Searching with early exit                             |

> **The flow-arch test:**
> Can you read what this code does without simulating how it runs?
> If yes — it's declarative enough.
> If no — extract a pure function, or replace the loop with a named transformation.
