# Tutorial 03 — Lazy Evaluation: Why `.map()` Is Eager and What To Do About It

> **flow-arch / flow-starter**
> Tutorial-02 taught you what breaks a pure function.
> This tutorial teaches you something subtler:
> code can be pure, declarative, and still algorithmically inefficient.
> The culprit is **eager evaluation** — and the fix is **lazy evaluation**.

---

## The problem: a chain that looks right but isn't

Start with this code. It is pure (assuming `factor` is a parameter).
It is declarative. It reads cleanly. And it has a hidden performance problem.

```javascript
const result = items
  .map((x) => x.values)
  .reduce((a, b) => a.concat(b), [])
  .map((v) => v * factor)
  .reduce((sum, v) => sum + v, 0);
```

To understand the problem, you need to understand one concept first.

---

## 1. Eager vs Lazy evaluation

|                        | Eager                               | Lazy                                       |
| ---------------------- | ----------------------------------- | ------------------------------------------ |
| **When work happens**  | Immediately, all at once            | Only when the result is actually needed    |
| **Memory**             | Full intermediate arrays created    | Values produced one at a time              |
| **Early exit**         | Impossible — all elements processed | Natural — stop when you have what you need |
| **JavaScript default** | ✅ All native array methods         | ❌ Not built-in — requires Generator       |

Every native JavaScript array method — `.map()`, `.filter()`, `.reduce()`, `.concat()` — is **eager**.

When you call `.map()`, it immediately:

1. Allocates a new array of the same length
2. Runs your callback on every single element
3. Returns the complete new array

Even if you only need the first result. Even if a later step will discard most of it.

---

## 2. Tracing the original code — step by step

Let `items` have **n elements**, each with an average of **m values**.
Total values across all items: **N = n × m**.

```javascript
const result = items
  .map((x) => x.values)
  // Step 1: O(n)
  // Iterates n items, creates an array of n arrays
  // Memory: O(n) array references

  .reduce((a, b) => a.concat(b), [])
  // Step 2: ⚠️  O(N²) — this is the hidden problem
  //
  // concat() creates a BRAND NEW array and copies everything each time:
  //
  // Iteration 1: []          + [v1, v2]  → new array [v1, v2]        (copies 2)
  // Iteration 2: [v1, v2]    + [v3]      → new array [v1, v2, v3]    (copies 3)
  // Iteration 3: [v1,v2,v3]  + [v4, v5] → new array [v1,v2,v3,v4,v5](copies 5)
  //
  // Total copies: 2 + 3 + 5 + ... → grows with N → O(N²)
  // Memory: O(N) final array + O(N) garbage from discarded copies

  .map((v) => v * factor)
  // Step 3: O(N) — one full pass, creates new array
  // Memory: O(N)

  .reduce((sum, v) => sum + v, 0);
// Step 4: O(N) — one full pass, returns a single number
// Memory: O(1)
```

**Total time complexity: O(N²)** — dominated entirely by the `.concat()` in Step 2.
**Total space complexity: O(N)** — multiple full intermediate arrays created and thrown away.

---

## 3. Fix 1 — Replace `.concat()` with `.flatMap()`

`flatMap` is a single built-in method that does `.map()` + flatten in **one pass**.
It does not repeatedly copy and reallocate. It builds the result array once.

```javascript
// ❌ Original — O(N²) due to concat in reduce
const result = items
  .map((x) => x.values)
  .reduce((a, b) => a.concat(b), [])
  .map((v) => v * factor)
  .reduce((sum, v) => sum + v, 0);

// ✅ Fixed — O(N) with flatMap
const result = items
  .flatMap((x) => x.values)
  .map((v) => v * factor)
  .reduce((sum, v) => sum + v, 0);
```

### Why `flatMap` solves the complexity problem

```javascript
// What flatMap does internally — one pass, no repeated copying
// Conceptually equivalent to (but engine-optimised):
const flatten = (items) => {
  const result = [];
  for (const item of items) {
    for (const value of item.values) {
      result.push(value); // push is amortized O(1) — no full copy
    }
  }
  return result;
};
```

`push` onto a growing array is **amortized O(1)** — the engine doubles
capacity when needed, so the average cost per element is O(1).

`concat` is always **O(current length)** — copies the entire array every time.

```
concat chain across N total values:
  copies: 1 + 2 + 3 + ... + N = N(N+1)/2  →  O(N²)

flatMap across N total values:
  pushes: N times, amortized O(1) each     →  O(N)
```

### Tracing the fixed version

```javascript
const result = items
  .flatMap((x) => x.values)
  // Step 1: O(N) — one pass, builds flat array once
  // Memory: O(N) — one array

  .map((v) => v * factor)
  // Step 2: O(N) — one pass
  // Memory: O(N) — one new array

  .reduce((sum, v) => sum + v, 0);
// Step 3: O(N) — one pass, produces one number
// Memory: O(1)

// Total time:  O(N) + O(N) + O(N) = O(N)  ✅
// Total space: O(N) intermediate arrays    (improvement, but still allocates)
```

**Time complexity drops from O(N²) to O(N).** This is a significant real-world improvement.

---

## 4. Fix 2 — Make it a named pure function (flow-arch style)

The chain is now efficient. One more step makes it fully flow-arch compliant.

```javascript
// ✅ Efficient, but intent is still slightly buried
const result = items
  .flatMap((x) => x.values)
  .map((v) => v * factor)
  .reduce((sum, v) => sum + v, 0);

// ✅✅ Efficient + named intent + all dependencies explicit
const extractValues = (item) => item.values;
const multiply = (factor) => (v) => v * factor; // curried pure function
const sum = (acc, v) => acc + v;

const sumScaledValues = (items, factor) =>
  items.flatMap(extractValues).map(multiply(factor)).reduce(sum, 0);

// Usage — explicit, testable, memoizable
const result = sumScaledValues(state.items, state.factor);
```

Each extracted function is independently testable in one line:

```javascript
expect(extractValues({ values: [1, 2, 3] })).toEqual([1, 2, 3]);
expect(multiply(2)(5)).toBe(10);
expect(sum(10, 5)).toBe(15);
expect(sumScaledValues([{ values: [1, 2] }, { values: [3] }], 2)).toBe(12);
// (1*2) + (2*2) + (3*2) = 2 + 4 + 6 = 12  ✅
```

---

## 5. Fix 3 — Generator lazy evaluation (zero intermediate arrays)

The `flatMap` version still creates **two intermediate arrays** that get discarded.

```
.flatMap()  →  full flat array in memory  (then discarded)
.map()      →  full scaled array in memory (then discarded)
.reduce()   →  reads the array, returns a number
```

If the dataset is very large, this is pure overhead.
You compute every value, store every value, then throw the stored arrays away.

**The ideal:** compute each value once, pass it forward immediately, never store intermediate results.

This is **lazy evaluation**. JavaScript's `Generator` makes it possible.

```javascript
// Lazy flatMap — yields one value at a time, never builds a full array
function* lazyFlatMap(iterable, fn) {
  for (const item of iterable) {
    yield* fn(item); // yield* delegates — spreads the inner iterable lazily
  }
}

// Lazy map — transforms one value at a time
function* lazyMap(iterable, fn) {
  for (const value of iterable) {
    yield fn(value);
  }
}

// Reduce must be eager — it inherently needs all values to produce a result
const lazyReduce = (iterable, fn, initial) => {
  let acc = initial;
  for (const value of iterable) {
    acc = fn(acc, value);
  }
  return acc;
};
```

### Composing the lazy pipeline

```javascript
const sumScaledValuesLazy = (items, factor) => {
  const flat = lazyFlatMap(items, extractValues); // no work done yet
  const scaled = lazyMap(flat, multiply(factor)); // no work done yet
  return lazyReduce(scaled, sum, 0); // work happens HERE
  // values flow through the pipeline one at a time
  // no intermediate array is ever fully materialised in memory
};
```

### What happens at runtime — step by step

```
items = [{ values: [1, 2] }, { values: [3, 4] }, { values: [5] }]
factor = 2

reduce requests first value from scaled:
  scaled requests from flat
  flat yields 1  (from items[0].values[0])
  scaled applies multiply(2)(1) → yields 2
  reduce: acc = 0 + 2 = 2

reduce requests second value:
  scaled requests from flat
  flat yields 2  (from items[0].values[1])
  scaled applies multiply(2)(2) → yields 4
  reduce: acc = 2 + 4 = 6

reduce requests third value:
  flat moves to items[1], yields 3
  scaled applies multiply(2)(3) → yields 6
  reduce: acc = 6 + 6 = 12

... continues until all values exhausted

Final result: 20
Peak memory: O(1) extra — no intermediate array ever existed
```

## 6. Comparing all approaches

```javascript
// Setup: 1000 items × 1000 values = 1,000,000 total values
const items = Array.from({ length: 1000 }, (_, i) => ({
  values: Array.from({ length: 1000 }, (_, j) => i * 1000 + j),
}));
const factor = 2;
```

| Approach                | Time complexity | Space complexity         | Style                 |
| ----------------------- | --------------- | ------------------------ | --------------------- |
| Original (`.concat()`)  | **O(N²)** ⚠️    | O(N) + garbage           | Declarative — broken  |
| `flatMap` chain         | **O(N)** ✅     | O(N) intermediate arrays | Declarative — good    |
| Generator lazy pipeline | **O(N)** ✅     | **O(1)** extra ✅        | Declarative — optimal |
| Nested `for` loop       | **O(N)** ✅     | **O(1)** ✅              | Imperative — optimal  |

**For most application code:** the `flatMap` chain is the right choice.
Readable, correct, and fast enough for hundreds to thousands of items.

**For large datasets:** the Generator pipeline matches `for` loop performance
while remaining fully declarative and composable.

---

## 7. When to use each approach

```javascript
// ✅ flatMap — normal application data
const result = items
  .flatMap(extractValues)
  .map(multiply(factor))
  .reduce(sum, 0);
// Use when: data fits comfortably in memory, readability is the priority

// ✅ lazy Generator — large or streaming data
const result = sumScaledValuesLazy(hugeItems, factor);
// Use when: data is large, memory is constrained, or you need early exit

// ✅ for loop — when lazy adds complexity without benefit
let total = 0;
for (const item of items) {
  for (const value of item.values) {
    total += value * factor;
  }
}
// Use when: the data structure is deeply nested or access patterns are irregular
```

---

## 8. The one rule that prevents O(N²) in declarative code

> **Never use `.concat()` inside `.reduce()` to flatten arrays.**
> Use `.flatMap()` instead.

This single rule eliminates the most common accidental O(N²) in declarative JavaScript.

```javascript
// ❌ This pattern — always O(N²)
array.reduce((acc, item) => acc.concat(item.values), []);

// ✅ This pattern — always O(N)
array.flatMap((item) => item.values);
```

---

## 9. How this connects to flow-arch

In flow-arch, every complexity decision should be **visible and intentional** — never accidental.

```
Transformation layer (default):
  .flatMap()  instead of  .map().reduce(concat)
  O(N) as the baseline — no accidental O(N²)

Transformation layer (large data):
  Generator lazy pipeline
  O(1) extra space — values stream through

Boundary layer (performance critical):
  for loop, pre-allocated arrays
  Imperative but contained — does not leak into the transformation layer

Never acceptable anywhere:
  .concat() inside .reduce() for flattening
  Accidental O(N²) from eager intermediate materialisation
  Complexity that is invisible to the reader
```

The goal is not to write the most clever code.
The goal is to write code where the **complexity is as obvious as the intent**.

---

## Summary

| Concept                        | What to remember                                                               |
| ------------------------------ | ------------------------------------------------------------------------------ |
| **Eager evaluation**           | All native JS array methods execute fully and immediately                      |
| **`.concat()` in `.reduce()`** | O(N²) — copies the entire array on every iteration. Never use this to flatten. |
| **`.flatMap()`**               | O(N) — the correct replacement. One pass, one allocation.                      |
| **`push` vs `concat`**         | `push` is amortized O(1). `concat` is O(n) every time.                         |
| **Generator lazy pipeline**    | O(1) extra space — values flow one at a time, no intermediate arrays           |
| **When to use flatMap**        | Default choice for flattening + transforming in application code               |
| **When to use lazy**           | Large datasets, memory pressure, streaming, or early-exit patterns             |
| **flow-arch rule**             | Complexity must be visible and intentional — never an accident                 |
