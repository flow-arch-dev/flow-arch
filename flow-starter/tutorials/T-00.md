# Tutorial 00 — Declarative Thinking & Pure Functions

> **flow-arch / flow-starter**
> This is the philosophical foundation of everything in flow-arch.
> Before you write a single component, understand this first.

---

## The core question

When you write code, you are always making a choice — consciously or not:

> **Am I telling the computer _how_ to do something?**
> **Or am I telling it _what_ I want?**

That choice is the difference between **imperative** and **declarative** style.

---

## 1. Imperative vs. Declarative

|                  | Imperative                         | Declarative                           |
| ---------------- | ---------------------------------- | ------------------------------------- |
| **Focus**        | Step-by-step process               | The desired result                    |
| **Control**      | `for` loops, `while`, manual state | `map`, `filter`, function composition |
| **Readability**  | Grows harder as logic grows        | Reads close to natural language       |
| **Mental model** | You manage _how_ it happens        | You describe _what_ should happen     |

### A real example: double the even numbers

**Imperative — the "how" way:**

```javascript
const numbers = [1, 2, 3, 4];
const result = [];

for (let i = 0; i < numbers.length; i++) {
  if (numbers[i] % 2 === 0) {
    result.push(numbers[i] * 2);
  }
}
```

You are manually managing:

- a loop counter `i`
- a mutable `result` array
- the push operation
- the index access `numbers[i]`

The _intent_ is buried inside the machinery. To understand what this does, you have to trace every step.

---

**Declarative + Pure — the flow-arch way:**

```javascript
const numbers = [1, 2, 3, 4];

// 1. Define Pure Functions (The "What")
const isEven = (n) => n % 2 === 0;
const double = (n) => n * 2;

// 2. Use Declarative Chaining
const result = numbers.filter(isEven).map(double);
```

No loop counter. No mutable array. No manual index.

You are _declaring_ a pipeline:

> "From `numbers`, keep the even ones, then double each."

Anyone reading this code understands the intent instantly — without tracing a single loop.

## 2. Arrow functions — why they look that way

You will see arrow functions throughout flow-arch. They are not just syntax sugar — they encourage a specific thinking style.

### Basic form

```javascript
// Traditional function
function double(n) {
  return n * 2;
}

// Arrow function — same thing
const double = (n) => n * 2;
```

### The `{}` and `return` rules

This is where arrow functions get precise:

**Without `{}` — implicit return (concise body):**

```javascript
const double = (n) => n * 2;
//                    ^^^^^^
//            this expression is automatically returned
//            no `return` keyword needed
//            no curly braces
```

Use this when your function is a **single expression**. This is the pure function ideal — one input, one output, no ceremony.

**With `{}` — explicit return required (block body):**

```javascript
const double = (n) => {
  const result = n * 2;
  return result; // now you must write return explicitly
};
```

Use this when you need multiple lines or intermediate variables.

**In the example code:**

```javascript
const isEven = (n) => n % 2 === 0;
//  No {}  →  no return needed
//  n % 2 === 0 is a single expression → it is returned directly
//  Result: true or false

const double = (n) => n * 2;
//  No {}  →  no return needed
//  n * 2 is a single expression → it is returned directly
//  Result: the doubled number
```

Both functions are **one line, one expression, one return value**. This is the purest form of a pure function.

---

## 3. Pure functions — the secret ingredient

A function is **pure** if it satisfies two rules:

### Rule 1 — Deterministic

> Same input → same output. Always.

```javascript
// Pure — always returns the same result for the same input
const double = (n) => n * 2;
double(4); // → 8, every single time

// Impure — result depends on something outside
let multiplier = 2;
const scaleBy = (n) => n * multiplier; // depends on external variable
multiplier = 3;
scaleBy(4); // → 12 now, not 8. The outside world changed the result.
```

### Rule 2 — No side effects

> The function does not touch anything outside itself.

```javascript
// Pure — returns a new array, does not touch the original
const doubled = numbers.map(double);
// numbers is unchanged

// Impure — modifies something outside its scope
const result = [];
function pushDoubled(n) {
  result.push(n * 2); // side effect: modifying an external array
}
```

Side effects include:

- Modifying a variable outside the function
- Logging to console (inside business logic)
- Making an API call (inside a transformation)
- Changing the DOM (inside a reducer or view function)

> In flow-arch, **side effects live at the boundary only** — in the Web Component shell, not inside reducers or view functions.

### Why pure functions make declarative code possible

Pure functions are predictable. Because they never surprise you, you can **compose** them safely:

```javascript
numbers
  .filter(isEven) // pure — returns a new array
  .map(double); // pure — returns a new array
```

Each step produces a new value. Nothing is mutated. The pipeline is self-documenting.

---

## 4. `.filter()` and `.map()` — how they work

These two array methods are the workhorses of declarative data transformation.

### `.filter(predicate)`

Takes a **predicate function** — a function that returns `true` or `false`.
Keeps only the elements where the predicate returns `true`.

```javascript
const numbers = [1, 2, 3, 4];

numbers.filter(isEven);
// isEven(1) → false → dropped
// isEven(2) → true  → kept
// isEven(3) → false → dropped
// isEven(4) → true  → kept
// Result: [2, 4]
```

The original array is **never modified**. `.filter()` always returns a new array.

### `.map(transform)`

Takes a **transform function** — a function that converts one value to another.
Applies the transform to every element and returns a new array of the same length.

```javascript
const evens = [2, 4];

evens.map(double);
// double(2) → 4
// double(4) → 8
// Result: [4, 8]
```

Again — the original array is never modified.

### Tracing the full example

```javascript
const numbers = [1, 2, 3, 4];

const result = numbers
  .filter(isEven) // [1,2,3,4] → [2,4]
  .map(double); // [2,4]     → [4,8]

// result = [4, 8]
// numbers = [1, 2, 3, 4]  ← completely unchanged
```

Three values exist at the end. None replaced the other. This is **immutability in practice**.

## 5. Why OOP makes this harder

Object-Oriented Programming groups data and behaviour together inside objects. This creates a specific set of problems that flow-arch deliberately avoids.

### The hidden state problem

```javascript
// OOP style
class Counter {
  constructor() {
    this.count = 0; // state lives inside the object
  }

  increment() {
    this.count++; // modifies internal state — a side effect
  }

  getCount() {
    return this.count; // output depends on when you call it
  }
}

const c = new Counter();
c.increment();
c.increment();
c.getCount(); // → 2
// But: what happened between construction and getCount()?
// You have to trace the entire object's history to know.
```

The output of `getCount()` depends on the history of mutations. It is not deterministic at call time. You cannot compose it safely.

### The inheritance coupling problem

```javascript
class Animal {
  speak() {
    return "...";
  }
}

class Dog extends Animal {
  speak() {
    return "Woof";
  }
}

class GoldenRetriever extends Dog {
  // What does speak() do here?
  // You have to trace the entire inheritance chain to know.
}
```

Inheritance creates tight coupling across layers. Changing a parent class can silently break children. The logic is distributed and hard to trace.

### The flow-arch alternative

```javascript
// flow-arch style — pure functions, no shared state
const createState = () => ({ count: 0 });

const reducer = (state, action) => {
  switch (action.type) {
    case "INCREMENT":
      return { ...state, count: state.count + 1 };
    default:
      return state;
  }
};
```

- `reducer` is a pure function — same state + same action → same new state, always
- No hidden history to trace
- No inheritance chain to follow
- Composable, testable, predictable by construction

The difference is not style. It is a different theory of where complexity should live.

---

## 6. How this connects to flow-arch

The example code in this tutorial is not just an exercise. It is the exact mental model that flow-arch applies at every layer:

```
// The flow-arch loop — the same idea, scaled up

Raw data                     →  filter(isValid)     →  Valid data
Valid data                   →  map(transform)      →  Processed data
Processed data               →  reduce(accumulate)  →  Final state
Final state                  →  view(state)         →  HTML string
Action + HTML string         →  reducer(state, action) →  New state
```

Every arrow is a pure function. Every step is declarative. Side effects — rendering to the DOM, reading from the network — happen only at the edges.

This is what flow-arch means by:

> **Declarative. Pure functions. No side effects.**

## Summary

| Concept                      | What it means in practice                                        |
| ---------------------------- | ---------------------------------------------------------------- |
| **Declarative**              | Describe what you want, not how to do it                         |
| **Pure function**            | Same input → same output, no side effects                        |
| **Arrow function (no `{}`)** | Single expression, implicit return — the purest form             |
| **`.filter()`**              | Keep elements matching a predicate, returns new array            |
| **`.map()`**                 | Transform every element, returns new array                       |
| **Immutability**             | Never modify — always return something new                       |
| **OOP problem**              | Hidden state and inheritance make code hard to trace and compose |
| **flow-arch solution**       | Pure reducers + pure views, side effects at the boundary only    |
