# Tutorial 06 — Loops Dissected: `for`, `for/in`, `while`, `do/while` and Their flow-arch Replacements

> **flow-arch / flow-starter**
> Every loop is a compressed description of state changes over time.
> This tutorial tears each one apart — what it really does, what cognitive cost it carries,
> and how flow-arch replaces it with something you can read, test, and compose.

---

## The shared dataset

```javascript
const users = [
  { id: 1, name: "Austin", score: 90, active: true, tags: ["js", "css"] },
  { id: 2, name: "Bob", score: 45, active: false, tags: ["python"] },
  { id: 3, name: "Carol", score: 78, active: true, tags: ["js", "haskell"] },
  { id: 4, name: "Dana", score: 55, active: true, tags: ["css", "elixir"] },
];
```

---

## Why loops carry hidden cost

Before the individual loops, a principle worth stating clearly:

Every loop forces the reader to do three things simultaneously:

```
1. Track state     — what variables are changing?
2. Track position  — where in the data are we?
3. Infer intent    — what is this loop actually trying to produce?
```

Declarative alternatives eliminate jobs 1 and 2 entirely.
The reader only has to do job 3 — understand the intent — which is the only one that matters.

## 1. `for` loop — the classic index loop

### Syntax

```javascript
for (initialiser; condition; update) {
  body;
}
```

### What the reader must track

```javascript
for (let i = 0; i < users.length; i++) {
  console.log(users[i].name);
}
```

| What the reader must hold in mind | Why it costs                                      |
| --------------------------------- | ------------------------------------------------- |
| `i` starts at 0                   | Initial state                                     |
| `i < users.length`                | When does this stop?                              |
| `i++`                             | How does state change each iteration?             |
| `users[i]`                        | Manual index access — could `i` be wrong?         |
| Body side effects                 | What does `console.log` do? Does it change state? |

Five simultaneous concerns for "print each user's name."

### Common patterns and their flow-arch replacements

**Pattern 1 — Iterate and collect**

```javascript
// ❌ for loop
const names = [];
for (let i = 0; i < users.length; i++) {
  names.push(users[i].name);
}
// names = ["Austin", "Bob", "Carol", "Dana"]

// ✅ flow-arch
const names = users.map((u) => u.name);
// ["Austin", "Bob", "Carol", "Dana"]

// Trace — what map does:
// u = users[0] → u.name = "Austin"  → collected
// u = users[1] → u.name = "Bob"     → collected
// u = users[2] → u.name = "Carol"   → collected
// u = users[3] → u.name = "Dana"    → collected
// Result: ["Austin", "Bob", "Carol", "Dana"]
```

**Pattern 2 — Filter and collect**

```javascript
// ❌ for loop
const active = [];
for (let i = 0; i < users.length; i++) {
  if (users[i].active) {
    active.push(users[i]);
  }
}

// ✅ flow-arch
const active = users.filter((u) => u.active);

// Trace:
// u = Austin (active:true)  → kept
// u = Bob    (active:false) → dropped
// u = Carol  (active:true)  → kept
// u = Dana   (active:true)  → kept
// Result: [Austin, Carol, Dana]
```

**Pattern 3 — Filter then transform**

```javascript
// ❌ for loop
const activeNames = [];
for (let i = 0; i < users.length; i++) {
  if (users[i].active) {
    activeNames.push(users[i].name.toUpperCase());
  }
}

// ✅ flow-arch — each step is a separate, named concern
const toUpperName = (u) => u.name.toUpperCase();
const isActive = (u) => u.active;

const activeNames = users.filter(isActive).map(toUpperName);
// ["AUSTIN", "CAROL", "DANA"]
```

**Pattern 4 — Accumulate a total**

```javascript
// ❌ for loop
let total = 0;
for (let i = 0; i < users.length; i++) {
  total += users[i].score;
}

// ✅ flow-arch
const total = users.reduce((acc, u) => acc + u.score, 0);

// Trace:
// acc=0,  u=Austin → 0  + 90 = 90
// acc=90, u=Bob    → 90 + 45 = 135
// acc=135,u=Carol  → 135 + 78 = 213
// acc=213,u=Dana   → 213 + 55 = 268
// Result: 268
```

**Pattern 5 — Flatten nested arrays**

```javascript
// ❌ nested for loop
const allTags = [];
for (let i = 0; i < users.length; i++) {
  for (let j = 0; j < users[i].tags.length; j++) {
    allTags.push(users[i].tags[j]);
  }
}
// Two loop variables, two levels of index access

// ✅ flow-arch
const allTags = users.flatMap((u) => u.tags);
// ["js", "css", "python", "js", "haskell", "css", "elixir"]

// Trace:
// u=Austin → yield "js", "css"
// u=Bob    → yield "python"
// u=Carol  → yield "js", "haskell"
// u=Dana   → yield "css", "elixir"
```

**Pattern 6 — Build a lookup object**

```javascript
// ❌ for loop
const byId = {};
for (let i = 0; i < users.length; i++) {
  byId[users[i].id] = users[i];
}

// ✅ flow-arch
const byId = users.reduce(
  (acc, u) => ({
    ...acc,
    [u.id]: u,
  }),
  {},
);
// { 1: Austin, 2: Bob, 3: Carol, 4: Dana }
```

**Pattern 7 — When `for` with index is still correct**

```javascript
// ✅ Keep for when you genuinely need the index for logic
// Example: zip two arrays together
const scores = [90, 45, 78, 55];
const bonuses = [5, 10, 3, 8];

// No clean Array method for this — for is appropriate
const combined = [];
for (let i = 0; i < scores.length; i++) {
  combined.push(scores[i] + bonuses[i]);
}
// [95, 55, 81, 63]

// Or use Array.from which handles the index cleanly:
const combined = Array.from(
  { length: scores.length },
  (_, i) => scores[i] + bonuses[i],
);
```

## 2. `for...of` loop — iterate over values

`for...of` is a cleaner version of the index `for` loop.
It removes the index variable but keeps the imperative structure.

### Syntax

```javascript
for (const item of iterable) {
  body;
}
```

### What it removes vs the index `for`

```javascript
// ❌ index for — manages i, condition, increment
for (let i = 0; i < users.length; i++) {
  console.log(users[i].name);
}

// ✅ for...of — no index, no condition, no increment
for (const user of users) {
  console.log(user.name);
}
```

`for...of` eliminates three of the five tracking concerns.
But the remaining two — **what state does the body change?** and **what is the intent?** — still apply.

### When for...of is the right choice

```javascript
// 1. When you need early exit (break)
for (const user of users) {
  if (user.score > 80) {
    console.log(`First high scorer: ${user.name}`);
    break; // stops immediately — cleaner than finding the index
  }
}
// ✅ Better as: users.find(u => u.score > 80)

// 2. Consuming a Generator (lazy pipeline)
function* activeUsers(users) {
  for (const user of users) {
    if (user.active) yield user;
  }
}

for (const user of activeUsers(users)) {
  await saveUser(user); // side effect — appropriate in boundary layer
}

// 3. Async iteration
for await (const line of readLines("file.txt")) {
  await processLine(line);
}
// ✅ This is the correct pattern — no declarative alternative without a library
```

### flow-arch position on `for...of`

```
Transformation layer:  Replace with .filter().map().reduce()
                       Use .find() for early exit

Boundary layer:        for...of is acceptable for:
                       — consuming Generators (side effects per item)
                       — for await...of with streams
                       — when break is needed and find/some are not sufficient
```

## 3. `for...in` loop — iterate over object keys

### Syntax

```javascript
for (const key in object) {
  body;
}
```

### The problem with `for...in`

```javascript
const user = { id: 1, name: "Austin", score: 90 };

for (const key in user) {
  console.log(key, user[key]);
}
// "id" 1
// "name" "Austin"
// "score" 90
```

This looks fine. But `for...in` has a hidden danger:

```javascript
// for...in iterates INHERITED properties too
function User(name) {
  this.name = name;
}
User.prototype.role = "member"; // added to prototype

const u = new User("Austin");

for (const key in u) {
  console.log(key);
}
// "name"   ← own property
// "role"   ← INHERITED — unexpected!

// The guard: always check hasOwnProperty
for (const key in u) {
  if (u.hasOwnProperty(key)) {
    console.log(key);
  }
}
// "name" only
```

This is why `for...in` on arrays is particularly dangerous:

```javascript
// ❌ Never use for...in on arrays
const arr = [10, 20, 30];
arr.customMethod = "oops"; // someone added a property to the array

for (const key in arr) {
  console.log(key);
}
// "0"
// "1"
// "2"
// "customMethod"  ← not an index — causes bugs
```

### flow-arch replacements for `for...in`

```javascript
const user = { id: 1, name: "Austin", score: 90 };

// Get all keys
Object.keys(user);
// ["id", "name", "score"]

// Get all values
Object.values(user);
// [1, "Austin", 90]

// Get key-value pairs
Object.entries(user);
// [["id", 1], ["name", "Austin"], ["score", 90]]

// Transform an object's values — pure
const doubled = Object.fromEntries(
  Object.entries(user).map(([key, val]) => [
    key,
    typeof val === "number" ? val * 2 : val,
  ]),
);
// { id: 2, name: "Austin", score: 180 }
// (name unchanged — not a number)

// Filter an object's keys — pure
const onlyNumbers = Object.fromEntries(
  Object.entries(user).filter(([_, val]) => typeof val === "number"),
);
// { id: 1, score: 90 }
```

### Complete replacement map

| `for...in` use case | flow-arch replacement                                 |
| ------------------- | ----------------------------------------------------- |
| Iterate keys        | `Object.keys(obj).forEach(...)`                       |
| Iterate values      | `Object.values(obj).map(...)`                         |
| Iterate key+value   | `Object.entries(obj).map(([k,v]) => ...)`             |
| Transform values    | `Object.fromEntries(Object.entries(obj).map(...))`    |
| Filter keys         | `Object.fromEntries(Object.entries(obj).filter(...))` |

```javascript
// ❌ for...in version — verbose and dangerous
const result = {};
for (const key in scores) {
  if (scores.hasOwnProperty(key)) {
    result[key] = scores[key] * 2;
  }
}

// ✅ flow-arch version — safe, pure, testable
const doubleScores = (scores) =>
  Object.fromEntries(Object.entries(scores).map(([k, v]) => [k, v * 2]));

expect(doubleScores({ a: 10, b: 20 })).toEqual({ a: 20, b: 40 });
// One line test — pure function
```

## 4. `while` loop — condition-driven loop

### Syntax

```javascript
while (condition) {
  body; // runs as long as condition is true
}
```

### What the reader must track

```javascript
let i = 0;
while (i < users.length) {
  console.log(users[i].name);
  i++;
}
```

| Tracking burden            | Detail                                            |
| -------------------------- | ------------------------------------------------- |
| Initial state              | `i = 0` — defined somewhere before the loop       |
| Termination condition      | `i < users.length` — when does this become false? |
| State mutation inside body | `i++` — could be missed, causing infinite loop    |
| Body side effects          | What else does the body do?                       |
| Ordering                   | Does the order of operations inside matter?       |

`while` is more dangerous than `for` because the state update (`i++`)
is **not syntactically enforced** — it can be placed anywhere in the body,
or accidentally omitted, causing an infinite loop.

### When `while` is the only right tool

```javascript
// 1. Stream reading — you don't know when it ends
async function* readLines(path) {
  const rl = readline.createInterface({ input: fs.createReadStream(path) });
  for await (const line of rl) yield line;
}
// while is inside the readline interface — contained, not visible to caller

// 2. Retry logic — run until success or limit
const withRetry = async (fn, maxAttempts = 3) => {
  let attempts = 0;
  while (attempts < maxAttempts) {
    try {
      return await fn(); // success — return immediately
    } catch {
      attempts++;
      if (attempts === maxAttempts) throw new Error("Max retries exceeded");
      await sleep(1000 * attempts); // exponential backoff
    }
  }
};
// while is contained inside withRetry — caller sees only a Promise

// 3. Polling — wait for condition
const waitUntil = (condition, interval = 100) =>
  new Promise((resolve) => {
    const check = () => (condition() ? resolve() : setTimeout(check, interval));
    check();
  });
// No while at all — recursive setTimeout is cleaner
```

### flow-arch replacements for `while`

```javascript
// Pattern: while + counter → for...of or recursion
// ❌ while
let i = 0;
const results = [];
while (i < users.length) {
  if (users[i].active) results.push(users[i].name);
  i++;
}

// ✅ flow-arch
const results = users.filter((u) => u.active).map((u) => u.name);

// Pattern: while + unknown termination → Generator
// ❌ while reading unknown-length data
let line;
while ((line = readLine()) !== null) {
  process(line);
}

// ✅ flow-arch — Generator wraps the while
function* lines(source) {
  let line;
  while ((line = source.readLine()) !== null) {
    yield line; // while is contained INSIDE the generator
  }
}
// Caller sees: for (const line of lines(source)) — no while visible
```

### The wrapping principle for `while`

```
while CAN exist in flow-arch.
But it must live inside a container:
  — inside a Generator (yields values one at a time)
  — inside a Promise wrapper (resolves when done)
  — inside a utility function (retry, waitUntil, poll)

The caller never sees a while loop.
The caller sees a lazy sequence, a Promise, or a simple function call.
```

## 5. `do...while` loop — executes at least once

### Syntax

```javascript
do {
  body; // runs first, THEN checks condition
} while (condition);
```

### The key difference from `while`

```javascript
// while — may never execute (if condition is false from start)
let i = 5;
while (i < 3) {
  console.log(i); // never runs — 5 < 3 is false
}

// do...while — always executes at least once
let j = 5;
do {
  console.log(j); // runs once — prints 5
  j++;
} while (j < 3); // 6 < 3 is false → stops after one run
```

### When do...while makes sense

The only scenario where `do...while` is genuinely the right tool:
**when the first execution must always happen regardless of condition.**

```javascript
// Classic use: input validation loop
// Must ask at least once, keep asking until valid
let input;
do {
  input = prompt("Enter a number > 0:");
} while (isNaN(input) || Number(input) <= 0);
// Guarantees at least one prompt
```

In Node.js / backend contexts, this pattern almost never appears.
It is primarily a UI/CLI pattern.

### flow-arch replacement

```javascript
// ❌ do...while for retry with first attempt
let result;
let attempts = 0;
do {
  result = await fetchData();
  attempts++;
} while (!result.ok && attempts < 3);

// ✅ flow-arch — recursive promise, caller sees nothing
const fetchWithRetry = async (fn, remaining = 3) => {
  const result = await fn();
  if (result.ok || remaining <= 1) return result;
  return fetchWithRetry(fn, remaining - 1);
};

const result = await fetchWithRetry(() => fetchData());
```

## 6. Side-by-side: every loop vs flow-arch

Same task — "find names of active users with score > 70, sorted descending" —
written in every loop style, then in flow-arch.

### Index `for`

```javascript
const result = [];
for (let i = 0; i < users.length; i++) {
  if (users[i].active && users[i].score > 70) {
    result.push(users[i].name);
  }
}
result.sort((a, b) => {
  const scoreA = users.find((u) => u.name === a).score;
  const scoreB = users.find((u) => u.name === b).score;
  return scoreB - scoreA;
});
// Tracking burden: i, result, sort, nested find inside sort
```

### `for...of`

```javascript
const result = [];
for (const user of users) {
  if (user.active && user.score > 70) {
    result.push(user.name);
  }
}
result.sort();
// Better — no index. Still mutable result, sort mutates.
```

### `while`

```javascript
let i = 0;
const result = [];
while (i < users.length) {
  const u = users[i];
  if (u.active && u.score > 70) result.push(u.name);
  i++;
}
result.sort((a, b) => b.localeCompare(a));
// Hardest to read — most tracking burden
```

### flow-arch

```javascript
// Named pure functions — each testable in one line
const isActive = (u) => u.active;
const isHighScorer = (u) => u.score > 70;
const toName = (u) => u.name;
const byScoreDesc = (a, b) => b.score - a.score;

const result = users
  .filter(isActive) // [Austin(90), Carol(78), Dana(55)]
  .filter(isHighScorer) // [Austin(90), Carol(78)]
  .sort(byScoreDesc) // [Austin(90), Carol(78)]  — already sorted here
  .map(toName); // ["Austin", "Carol"]

// Trace:
// filter(isActive):      Austin✓ Bob✗ Carol✓ Dana✓  → [Austin,Carol,Dana]
// filter(isHighScorer):  Austin(90>70)✓ Carol(78>70)✓ Dana(55>70)✗ → [Austin,Carol]
// sort(byScoreDesc):     90 > 78 → [Austin, Carol]
// map(toName):           ["Austin", "Carol"]
```

**What changed:**

- No `i`, no `result = []`, no `push`, no `sort` mutation
- Each step named and independently testable
- Reader sees **what** happens, not **how**

## 7. The cognitive load comparison

```
Index for:   track i + condition + update + body + potential side effects
for...of:    track variable + body + potential side effects
while:       track condition + body + state update location + infinite loop risk
do...while:  same as while + "always runs once" mental model

filter:      one concern — does this item pass?
map:         one concern — what does this item become?
reduce:      one concern — how does this item change the accumulator?
find:        one concern — is this the item I want?
```

Declarative methods enforce **one concern per step**.
Loops allow — and often accumulate — multiple concerns per block.

## 8. Complete replacement map

| Loop pattern          | flow-arch replacement                | Notes                    |
| --------------------- | ------------------------------------ | ------------------------ |
| `for` collect → array | `.map()`                             | Always                   |
| `for` filter → array  | `.filter()`                          | Always                   |
| `for` accumulate      | `.reduce()`                          | Always                   |
| `for` find first      | `.find()`                            | Early exit built in      |
| `for` flatten nested  | `.flatMap()`                         | O(N) not O(N²)           |
| `for` build object    | `.reduce()` with `{}`                | Or `Object.fromEntries`  |
| `for` zip two arrays  | `Array.from({length}, (_,i) => ...)` | Index genuinely needed   |
| `for...in` keys       | `Object.keys().map()`                | Safer, no prototype leak |
| `for...in` values     | `Object.values().map()`              | Safer                    |
| `for...in` entries    | `Object.entries().map()`             | Safer                    |
| `while` collection    | `.filter().map()`                    | Always                   |
| `while` stream        | `async function*` + `for await`      | Generator wraps while    |
| `while` retry         | Recursive Promise                    | While hidden inside      |
| `while` poll          | `waitUntil()` wrapper                | While hidden inside      |
| `do...while` retry    | Recursive async function             | Cleaner                  |
| `do...while` prompt   | `while` or recursive fn              | Rare in backend          |

## 9. The flow-arch loop rule — complete statement

```
A loop is acceptable when:
  1. It is wrapped inside a Generator, Promise, or utility function
  2. The caller cannot see that a loop is happening
  3. It lives in the boundary layer, not the transformation layer
  4. There is no declarative alternative that expresses the same intent more clearly

A loop must be replaced when:
  1. It is inside a transformation function that should be pure
  2. It modifies external state (push to outer array, mutate outer variable)
  3. A named declarative alternative (filter/map/reduce/find) exists
  4. The intent of the loop can be expressed as a data transformation
```

## Summary

| Loop          | Core problem                                       | flow-arch position                      | When still valid                             |
| ------------- | -------------------------------------------------- | --------------------------------------- | -------------------------------------------- |
| `for` (index) | Manages i, condition, update — 5 tracking concerns | Replace with map/filter/reduce/find     | Zipping arrays, index-dependent logic        |
| `for...of`    | Cleaner than index for, still imperative           | Replace in transformation layer         | Boundary layer with side effects, generators |
| `for...in`    | Iterates inherited properties — dangerous          | Replace with Object.keys/values/entries | Almost never                                 |
| `while`       | State update can be anywhere — infinite loop risk  | Wrap in Generator or Promise            | Streams, retry, polling — always wrapped     |
| `do...while`  | Executes at least once — rare need                 | Recursive async function                | UI prompts, extremely rare in backend        |
